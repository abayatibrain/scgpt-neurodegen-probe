# ADR-0005: Negative-result framing
Date: 2026-05-19
Status: Proposed — awaiting Armin sign-off

## Context

The brief (§6.4 item 6) is unusually direct: **"If the model performs
at chance, the README says so plainly. This is the most important
reasoning checkpoint — the temptation to overclaim is real."**

Single-cell foundation models on neurodegeneration tasks have a real
literature of overclaiming, partly because the field has incentives
to publish positive results and partly because the metrics used in
many papers are easy to game (e.g., zero-shot cell-type
classification on data the model has effectively seen). This ADR
commits, in writing and before any benchmark is run, to a posture
on negative results that the README will not retreat from.

This is a defensibility decision more than a methodology decision —
the methodology is largely set by ADRs 0003 and 0004. The question
here is what the README does when the results disappoint.

## Options considered

### Option A — Plain-language statement in the first paragraph
If a benchmark comes up at chance (within CI of the random
baseline), the README's first paragraph names it plainly:
> "On Task A (PD-gene ranking), scGPT performed at chance
> (AUC 0.51, 95% CI 0.46-0.56; p = 0.41). The model's
> embedding does not encode PD-gene membership over what we
> would expect from a random gene set of the same size."

### Option B — Negative results in the "Limitations" section only
Keep the first paragraph optimistic; put any negative results in
the section everyone scrolls past.
- Pros: gentler on a hiring-manager skim.
- Cons: the brief explicitly forbids this posture; it's the
  overclaim pattern in published form.

### Option C — Frame negative results as "we found the limit"
Rhetorical reframe — chance performance is described as
"identifying the boundary of what current foundation models can do."
- Pros: salvages narrative.
- Cons: still overclaim. The README is not a literature paper;
  it's a portfolio piece evaluated by people who will see
  through the rhetorical reframe.

### Option D — Negative results in their own README section,
**before** the discussion of positive results
A "What this probe did not find" section appears in the README
*above* any "what it did find" discussion.
- Pros: rare in the field; makes the candidate look unusually
  honest; matches portfolio audience values (Sanofi / J&J
  reviewers know the literature has a credibility problem).
- Cons: requires the README to commit to that ordering
  irrespective of which benchmarks turned out positive.

## Criteria for choosing
1. **Brief compliance.** §6.4 item 6 explicitly requires plain
   statement of negative results.
2. **Portfolio defendability.** A hiring panel that sees a
   candidate honestly report a negative result trusts the
   positive results more.
3. **Self-consistency.** Whatever posture this ADR commits to
   must be enforceable: if the benchmark results are all
   positive, the README is still bound by this framing decision
   not to over-claim.

## Decision

**Option D: a "What this probe did not find" README section that
appears before the positive-result discussion.**

Specifically:

- The README's structure (per §2.2 of the brief) opens with the
  biological question, then the architecture diagram, then the
  results. The results section is split into **two named
  subsections**, in this order:
  1. **"What this probe did not find."** Every benchmark that
     came back at chance (within CI of the random baseline) is
     reported here, by name, with the headline number from
     ADR-0004.
  2. **"What this probe did find."** Every benchmark that came
     back above chance is reported here.

- If all benchmarks are positive, the "did not find" section
  contains a single sentence: "No benchmarks in this probe came
  back at chance." It is not omitted.

- If all benchmarks are negative, the "did find" section
  contains a single sentence: "No benchmarks in this probe came
  back above chance." The README's first paragraph leads with
  the negative result, plainly.

- The "Limitations" section (per §2.2) is **separate** from these
  two subsections. It enumerates structural reasons the probe
  could be wrong (training-data contamination risk, ground-truth
  curation choices, etc.), not negative results.

## Consequences
- The two-section results structure is committed in the README
  template before any benchmarks run. Reviewers can verify that
  the framing predates the results.
- The "did not find" section's existence is itself a credibility
  signal for portfolio reviewers, separate from the result
  content.
- The CI bands from ADR-0004 are the discriminator between "at
  chance" and "above chance" — a result is in the "did find"
  section if and only if its CI lower bound exceeds the random
  baseline.
