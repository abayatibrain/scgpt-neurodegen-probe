# ADR-0004: Reporting standards (effect sizes, CIs, p-value floor)
Date: 2026-05-19
Status: Proposed — awaiting Armin sign-off

## Context

The brief (§6.4 item 5) is explicit on what reporting should look
like: "Effect sizes and CIs over p-values. If you must report
p-values, never <0.001 (report exact below that)." This ADR
operationalises that guidance for every benchmark this probe runs.

The choice matters because foundation-model probes on
neurodegeneration tasks have a documented history of overstating
results via small effect sizes accompanied by impressive-looking
p-values. The reporting format itself is part of the defendability
of the probe's conclusions.

## Options considered

### Option A — Effect size + 95% bootstrap CI as the headline
Every result table reports:
- Point estimate (AUC, silhouette, macro-F1, etc.)
- 95% CI from bootstrap (n_boot = 1000) over the query set.
- Empirical p-value from gene-label permutation (ADR-0003).
- P-values are reported as `p < 0.001` if smaller than 1/n_perm;
  exact otherwise.

### Option B — Permutation p-value only
Just the empirical p-value, no CI.
- Pros: simplest.
- Cons: brief explicitly forbids this; p-values without effect
  sizes communicate the wrong thing.

### Option C — Effect size only, no p-value
Just the AUC + CI; do not report a p-value at all.
- Pros: forces the reader to interpret effect size; sidesteps
  multiple-testing concerns.
- Cons: a reviewer will still ask "is this above chance?" Having
  no answer is worse than having a properly-floored one.

### Option D — Full Bayesian (posterior over the metric)
Probabilistic ranking framework.
- Pros: arguably the most principled.
- Cons: pulls in PyMC/Stan; substantial implementation lift;
  difficult to defend to a reviewer who hasn't read the priors.

## Criteria for choosing
1. **Brief compliance.** §6.4 item 5 names effect sizes + CIs as
   the headline and floors p-values at 0.001.
2. **Readability for a non-specialist reviewer.** A hiring panel
   scanning the README needs to understand the result in seconds.
3. **Consistency across the three benchmarks.** Task A
   (PD-ranking), Task B (disease separation), Task C (cell-type
   classification) should all report in the same format so the
   README's results table is uniform.

## Decision

**Option A: effect size + 95% bootstrap CI as headline; empirical
p-value reported alongside; p-value floored at 0.001.**

Specifically:

- **Headline row in each results table:**
  > AUC 0.62 (95% CI 0.55–0.69); p < 0.001 (permutation, n=1000)

- **CI methodology:** bootstrap over the query set (e.g., the
  held-out PD-gene labels for Task A; the held-out cells for Task
  C), `n_boot = 1000`. Seed recorded in each PNG sidecar.

- **P-value floor:** if the empirical p-value would round to
  `0.000`, report it as `p < 0.001`. Never report `p = 0`.

- **Effect-size interpretation guide:** the README includes a
  short table noting which effect-size ranges count as "strong"
  vs "weak" for each benchmark, so the headline number has a
  reference for the reader. (E.g., AUC 0.5 is chance; 0.6 is
  weak; 0.7 is substantial; 0.8+ is strong.)

- **Multiple-testing posture:** three benchmarks × two models
  (Option D in ADR-0002) = six headline statistics. The README
  reports all six honestly; a Bonferroni-adjusted alpha = 0.05/6
  ≈ 0.008 is mentioned in the Method section as a conservative
  reader's threshold but not used to filter the displayed results.

## Consequences
- `stats/reporting.py` owns the formatter that produces the
  headline-row string from a `(point_estimate, ci, p_value, n_perm)`
  tuple. The format is the source of truth.
- The bootstrap implementation uses scikit-learn's
  `resample` with replacement; deterministic seed per benchmark.
- The "what this does not show" section in the README
  (per the existing scaffold) explicitly enumerates the multiple-
  testing concern.
