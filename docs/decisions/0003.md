# ADR-0003: Null distribution methodology
Date: 2026-05-19
Status: Proposed — awaiting Armin sign-off

## Context

Every benchmark in this probe (PD-gene ranking, disease separation,
cell-type classification) reduces to: "does the model's embedding
score the PD-relevant signal above what we would expect by chance?"
That phrase — "by chance" — is the null distribution. Choosing it
wrong is the most common way single-cell-foundation-model probes
overstate or understate their results.

The brief (§6.4 item 4) is explicit: "Permute gene labels, not
cell labels. Explain why."

## Options considered

### Option A — Permute gene labels
Shuffle which genes are tagged "PD-associated." The cell embedding
is fixed; the labelling is randomised. Repeat n_perm times; the
distribution of "PD-ranking AUC" under permuted labels is the null.
- Pros: tests what we actually want — "does the model encode the
  specific PD-genes' signal, or would any equal-sized random gene
  set give the same answer?" Preserves cell-embedding structure
  exactly.
- Cons: requires careful definition of "PD-associated" (ADR-0001
  handles this); if the PD-gene set is imbalanced in some
  uncontrolled property (e.g., gene length, expression level), the
  null can be biased.

### Option B — Permute cell labels
Shuffle which cells are PD vs control. The gene labelling is fixed.
- Pros: tests "does the model distinguish PD from control cells?"
- Cons: this is **the wrong question** for the headline benchmark.
  The headline benchmark is "do PD genes rank higher than random
  genes in the model's embedding of DA neurons?" Permuting cell
  labels would invalidate that ranking framing.

### Option C — Bootstrap PD-gene set membership
Resample the PD-gene set with replacement; measure ranking AUC
distribution.
- Pros: gives a confidence interval directly on the AUC.
- Cons: this is a CI on the observed AUC, not a null distribution.
  It doesn't tell us if the AUC is non-random — it tells us how
  uncertain our AUC estimate is.

### Option D — Match permutation on a covariate
Permute gene labels but restrict the permutation to genes matched
on expression level or gene length to control for known confounders.
- Pros: addresses the imbalance concern from Option A.
- Cons: requires choosing the matching covariate; introduces a
  further parameter to defend.

## Criteria for choosing
1. **Match to the question.** The benchmark question is about
   *gene*-level signal in the embedding; the null must be over
   gene labels.
2. **Pretrained-cutoff hygiene.** ADR-0001 already requires a
   held-out atlas postdating each model's cutoff. The null must
   not silently reintroduce training-data leakage.
3. **Defendability.** A reviewer should be able to read the null
   construction in two sentences and agree it tests the right
   thing.
4. **Computational tractability.** n_perm = 1000 is the minimum
   for stable AUC-distribution shape; we want this run-time
   bounded.

## Decision

**Option A: permute gene labels.** Per the brief's explicit
guidance and the criteria above. Specifically:

- For each benchmark, sample `n_perm = 1000` permutations of the
  PD-gene labels among the full set of genes the model's embedding
  covers.
- For each permutation, compute the same benchmark statistic (AUC,
  silhouette, or classification F1) as for the observed labelling.
- The empirical p-value is the fraction of permutations whose
  statistic equals or exceeds the observed.

Option D (covariate-matched permutation) is **deferred** to a
later ADR. The reasoning: if the headline result is positive
*without* covariate matching, the matched-permutation null would
only strengthen the conclusion; if the headline result is
negative, matched-permutation is unlikely to flip it.

The bootstrap CI (Option C) is **also reported alongside**, framed
explicitly as a CI on the point estimate rather than as a null.
ADR-0004 (reporting standards) defines the precise reporting format.

## Consequences
- Each benchmark module owns its own permutation routine and seed
  policy; seeds recorded in result PNG sidecar files for
  reproducibility.
- `n_perm = 1000` is a configurable threshold in `config.py`; the
  default value is documented as the floor below which p-value
  resolution becomes too coarse to report at <0.001.
- Empirical p-values below 1/n_perm are reported as `p < 0.001`
  (the floor); never as exact values below that, per ADR-0004.
